# Adding Custom Revival Achievements

Every revival achievement is two linked pieces of data, both generated by one script:

1. **A definition** in `src/main/resources/assets/hardcore_achievements_plus/revival_achievements.json` — what the panel and chat show, which tier it belongs to, and which vanilla advancements gate it.
2. **A backing advancement** in `src/main/resources/data/hardcore_achievements_plus/advancement/revival/<id>.json` — a real vanilla advancement whose completion triggers the revival. It also renders in the game's advancements screen under the mod's tab.

They are linked by the definition's `backing_advancement` id. At server start the loader validates every entry (backing file exists, `requires` ids resolve, icon is a real item) and disables broken ones with a log line — a bad entry never crashes the server.

> **Never hand-edit the generated JSONs.** Both files are written by `scripts/gen_achievements.py`. Edit the `ROSTER` list there and rerun:
>
> ```sh
> python3 scripts/gen_achievements.py
> ./gradlew build
> ```

## Step by step: adding an achievement

1. **Design the task.** One sentence: *what must a living player do?* Check it against the [design rules](#design-rules) — most ideas that fail do so because they need kill-counting (not expressible) or a biome-lottery item.

2. **Pick the tier and the gate.** Decide which progression stage the task belongs to (0 Embers → 4 Inferno), then pick a vanilla advancement that *proves* a player has reached that stage — that id goes in `requires`. Use the [vanilla reference tables](#vanilla-advancement-reference) below or the interactive [advancement atlas](advancement-atlas.html); the tier chip on the gate should match your entry's tier.

3. **Copy a template.** [`achievement-template.py`](achievement-template.py) has a fully-commented blank entry plus one worked example per criteria type (items-with-counts, kills, checklists, breeding/taming/summoning, biomes, bare triggers, vanilla-criteria reuse). Copy the closest block into the `ROSTER` list in `scripts/gen_achievements.py`.

4. **Fill in the fields** — id, title, mission-statement description, icon, tier, criteria, `requires`, `repeatable`. The field reference and helper gallery in the next two sections cover every option.

5. **Regenerate.** `python3 scripts/gen_achievements.py` rewrites the definitions file and all backing advancements (and asserts your id is unique). Then `./gradlew build`.

6. **Verify at startup.** Start the dev server (`./gradlew runServer`) and look for `Revival achievements active: N/N` — the count should include your entry. If it was disabled instead, the log line names the problem (unknown `requires` id, missing backing advancement, bad icon).

7. **Test in game.** In a hardcore dev world: check your entry appears in the advancements tab (L) and the H panel's Available section with the right lock state. To test the full loop quickly, have one player die, then complete the task — or shortcut the completion with:

   ```
   /advancement grant <player> only hardcore_achievements_plus:revival/<id>
   ```

   The grant command goes through the same code path as a real completion, so it revives the fallen player, records the completion, and (for repeatables) re-arms the backing advancement.

## The `ach(...)` entry

```python
ach("skeletal_reaper", "Skeletal Reaper",
    "Kill a wither skeleton and steal back a life from death itself",
    "minecraft:wither_skeleton_skull",     # icon (must be a real item id)
    3,                                     # tier 0-4
    kill("killed_wither_skeleton", "minecraft:wither_skeleton"),
    requires=["minecraft:nether/find_fortress"],
    repeatable=True),                      # default True; False = one-time, retired after use
```

| Field | Meaning |
|---|---|
| `id` | Stable snake_case identifier. Renaming/removing one later is safe — pending revivals referencing a removed id are reassigned at server start. |
| `title` / `description` | Shown in chat (with hover tooltip), title cards, the H panel, and the advancements tab. The description should say exactly what to do. |
| `icon` | Item id rendered in the panel and the advancements tab. |
| `tier` | 0 Embers · 1 Sparks · 2 Flames · 3 Blaze · 4 Inferno. Selection rolls in the highest eligible tier (20% chance of one tier down). |
| `requires` | Vanilla advancement ids — the difficulty gate. The achievement is only assignable when at least one *living* player has all of them. Empty = always eligible. |
| `repeatable` | `True`: progress is revoked after completion so it can be assigned again. `False`: retired forever after one use. |

## Criteria helpers

Criteria define the actual task inside the backing advancement. All criteria in an entry are **required** (checklist semantics). Helpers in the script:

| Helper | Task | Example |
|---|---|---|
| `inv(name, *item_pred(...))` | Hold items **simultaneously** | `inv("ores", item_pred("minecraft:raw_iron", 16), item_pred("#minecraft:wool", 32))` — supports `count` minimums and `#tags` |
| `kill(name, entity)` | Kill one entity of a type | `kill("kill_blaze", "minecraft:blaze")` |
| `breed(name, entity)` | Breed an animal type | `breed("breed_cow", "minecraft:cow")` |
| `tame(name, entity)` | Tame an animal type | `tame("tame_horse", "minecraft:horse")` |
| `summon(name, entity)` | Summon (golem, wither...) | `summon("summon_golem", "minecraft:iron_golem")` |
| `biome(name, biome)` | Set foot in a biome | `biome("deep_dark", "minecraft:deep_dark")` |
| `bare(name, trigger)` | Any condition-less trigger | `bare("brewed", "minecraft:brewed_potion")` |
| `vanilla_criteria("tab/slug")` | Copy a vanilla advancement's criteria verbatim | `vanilla_criteria("nether/ride_strider")` — for complex predicate shapes |
| `merge(a, b, ...)` | Combine into a multi-part checklist | `merge(kill(...), inv(...))` |

## Design rules

- **The engine cannot count kills.** "Kill 10 blazes" is not expressible; one criterion fires once. Grind comes from item `count` predicates, multi-criteria checklists, or kill+gather combos.
- **`inv` is hold-at-once.** All listed items must be in the inventory simultaneously — that is part of the difficulty.
- **Keep tier 0 safe.** At least one tier-0 entry must stay `repeatable` with empty `requires` — it is the fallback when nothing else is eligible. Ship several so fallback assignments stay varied.
- **Repeatable tasks must be redoable.** After completion the backing progress is revoked; avoid criteria a player can't naturally trigger again.
- **Avoid biome-lottery items** (jungle-only, badlands-only...) unless the tier justifies a long expedition — an unlucky world shouldn't make a revival impossible.
- **Pick the gate from real progression.** The `requires` id should prove the group has reached the stage the task demands — the tables below give every vanilla advancement with a suggested tier.

## Vanilla advancement reference

Every vanilla advancement in Minecraft 26.2, rated with a suggested revival tier. Use the **id** in `requires`; the **triggers** column tells you what mechanics its criteria use (copyable with `vanilla_criteria("tab/slug")`). The interactive version of this reference — [advancement-atlas.html](advancement-atlas.html) — also shows the unlock trees.

Tier scale: **T0** Embers (bare hands) · **T1** Sparks (iron age) · **T2** Flames (established base) · **T3** Blaze (Nether) · **T4** Inferno (End game).
